The Approach
6 Phases — Per Slice

Click each phase to expand. Follow this order, every time, for every slice.

0
Slice Selection & Baseline Capture
⚠️ Most skipped. Most critical. Do before any code.
▾

You are building the answer key before the exam. Everything captured here becomes your Design Eval criteria in Phase 1.

  • ✓ Document exactly what the old system does — behavior, not just code
  • ✓ Record API request/response shapes via Postman / HAR files
  • ✓ Screen-record exact user flows in the old system
  • ✓ Extract business rules from old controllers and services
  • ✓ Document edge cases: null inputs, empty states, error states
  • ✓ Capture performance baseline — page load time, API response time
  • ✓ Note DB state before and after each action

2
Migrate the Slice — Copilot Does the Heavy Lifting
🔨 Order matters. Never migrate FE and BE simultaneously.
▾

Always stable, verified backend before UI is built on top of it.

1st
Migrate BE Service Layer
→ Run Layer 2 evals immediately
2nd
Migrate .NET Core API Endpoint
→ Run Layer 1 evals. Old & new run in parallel.
3rd
Migrate Angular Component
→ Run Layer 3 evals. Point to new API.

⚡ Why this order? You always want a stable verified backend before you build UI on top of it. Migrating FE and BE simultaneously means you can't isolate where bugs came from.

⚠️ If You Skip Evals
6 Risks That Will Explode

These don't announce themselves. They hide. And surface at the worst possible moment.

01 🕵️
Silent Business Logic Corruption

Business rules migrate incorrectly but look correct on the surface. The code compiles. Tests pass. CI/CD is green. But the order of operations flipped.

On a ₹10L invoice, a discount-tax ordering flip = ₹15,000 difference. 400 invoices. 6 weeks. Nobody notices.
02 🔍
No Rollback Evidence

Something breaks in production. Your team tries to figure out which module caused it and when. No baseline. No comparison. No paper trail.

MTTR explodes. A 2-hour fix becomes a 2-day investigation.
03 🤖
Copilot Confidence Trap

Copilot generates clean, well-structured code that looks right. But it doesn't know your business. It doesn't know the 2019 workaround that never got documented.

You shipped AI-generated code that your team approved visually. Nobody tested outcomes. Liability sits with you.
04 ❄️
Regression Snowball

Sprint 1: 10 modules, no evals. Sprint 2: 10 more. A bug appears. Is it sprint 1 or sprint 2? Without evals per module, you can't isolate where regression entered.

You're re-testing everything from scratch. Sprint velocity collapses. Deadline slips.
05 😰
Client Acceptance Risk

UAT arrives. Client finds 40 defects. Your team scrambles. The client loses confidence. Every screen they test, they're now suspicious.

UAT becomes adversarial instead of confirmatory. Relationship damage. Potential penalty clauses triggered.
06 📊
No Measurement of AI Quality

You used Copilot to accelerate migration. Leadership asks: "How much did it actually help? Can we trust it on the next engagement?" Without evals, you have no data.

You lose the ability to prove ROI on your AI investment. You can't improve the process for the next project.
Manager View
Every Concern, Addressed
Manager Concern How This Approach Addresses It
"How do we know it works?" Eval results are the evidence, not engineer opinion. Every slice has a documented pass report before sign-off.
"What if Copilot gets it wrong?" Evals catch it before it merges — not after it ships. The pipeline is the safety net.
"How do we report progress?" Slices completed + eval pass rates = objective, reportable status. Not vibes.
"What's our rollback plan?" Each slice is independently verified. Rollback is clean and scoped — not a full system revert.
"How long will this take?" Slice velocity becomes predictable after slice 2-3. You build a reliable forecast, not a guess.
"Can we prove Copilot's ROI?" First-pass accuracy rate per module type, time saved vs time spent fixing — all measured, all reported.
We are not migrating a codebase.
We are migrating verified behavior,
one slice at a time.

The difference is — at the end, we can prove it works. Not hope it works.
